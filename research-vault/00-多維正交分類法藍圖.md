# 🛡️ 程式碼安全知識庫：雙層多維正交分類法藍圖 (V5-Final)

本文件記錄了專案知識庫分類系統的設計藍圖。為了解決傳統單一階層式分類（Hierarchical Taxonomy）在歸檔邊界模糊、多屬性交叉論文歸類困難等痛點，本知識庫採用**「雙層多維正交分類法（Dual-Layer Faceted Taxonomy）」**。

---

## 1. 雙層架構設計理念

本知識庫將「資訊檢索的分類正交性（維度篩選）」與「學術研究的精細顆粒度（同儕聚類）」分離，採用兩層並存結構：

1. **一級 Facet（四維正交維度）**：
   定義為筆記 Frontmatter 中的元數據（Metadata）屬性。這四個維度在邏輯上完全獨立且互斥，僅用於全局多維篩選與交叉查詢，**不對應物理資料夾**。
2. **二級細分標籤（39個精細複合流派節點）**：
   定義為知識庫中實際存在的 **39個物理流派筆記（.md 檔案）**。論文筆記透過雙括號 `[[2B.5-自動化測試生成]]` 直接連結到這些節點，用於 Obsidian 關係圖譜與 Dataview 的局域自動聚合。

## 2. 一級 Facet：四維正交維度定義與判定規約

### 📐 維度一：分析方法論 (Methodology - How)
*核心問句：此技術的核心程式分析理論與運作模型是什麼？（為維護語法一致性，一律使用「分析機制/運作型態」命名）*

1. **靜態分析 (Static Analysis)**
   - *正面定義*：不執行程式碼，分析其結構、語意或邏輯屬性。
   - *排除性定義*：排除任何需要真實執行（Concrete Execution）或虛擬 CPU 模擬執行（如 Unicorn/QEMU）的技術（該類歸入動態分析）。
2. **動態分析 - 模糊測試 (Dynamic - Fuzzing)**
   - *正面定義*：自動產生輸入並執行程式，以**觸發崩潰 (Crash) 或偵測嚴重異常**為核心目標。
   - *排除性定義*：排除以「生成可讀單元測試套件且主要以功能 Assert 驗證 Correctness」為目標的工具（該類歸入 ATG），以及使用約束求解器進行分支求解的技術（該類歸入符號執行）。
3. **動態分析 - 自動化測試生成 (Dynamic - ATG)**
   - *正面定義*：自動生成結構化測試套件（如 JUnit/Pytest），著重於**程式碼覆蓋率與生成有效斷言 (Assertions)**。
   - *排除性定義*：排除無斷言、僅以探測系統異常崩潰為目的的隨機/變異生成工具（該類歸入 Fuzzing）。
4. **動態分析 - 符號與混合執行 (Dynamic - Symbolic / Concolic)**
   - *正面定義*：使用符號值作為輸入，執行程式時收集路徑約束，並**利用 SMT/SAT 求解器生成新輸入**。
   - *排除性定義*：排除僅進行具體數值隨機/變異、無路徑約束求解（No constraint solving）的技術（該類歸入 Fuzzing/ATG）。
5. **動態分析 - 動態追蹤與監控 (Dynamic - Trace & Monitor)**
   - *正面定義*：透過 DBI、編譯期插樁或虛擬化層，在執行期收集執行路徑、內存狀態或變數污染傳播（DTA），以**偵測特定違規行為或收集執行軌跡**為目標（如 Sanitizers, RASP）。
   - *排除性定義*：排除本身不注入監控/插樁邏輯、亦不收集動態執行狀態，僅作為外部黑箱流量發送的掃描器（該類歸入黑箱/DAST）。
6. **動態分析 - 差異與變異測試 (Dynamic - Differential / Mutation)**
   - *正面定義*：對多個等價實現輸入相同資料以比對輸出差異（Differential），或變異代碼本身（Mutation）以評估測試套件強度。
   - *排除性定義*：排除只針對單一特定實現生成單元測試套件且有預期 assert 輸出的技術（該類歸入 ATG）。
7. **動態分析 - 黑箱與漏洞掃描 (Dynamic - Black-box / DAST)**
   - *正面定義*：在不獲取原始碼或程式內部狀態的黑盒情況下，從外部發送請求/資料，僅**分析其外部響應（如狀態碼、內容、耗時）以識別漏洞**。
   - *排除性定義*：排除任何在二進位/字節碼層注入檢測邏輯或監視內部污染傳播的技術（該類歸入動態追蹤與監控），以及以自動變異輸入探索代碼覆蓋率或尋找底層崩潰為核心目標的技術（該類歸入 Fuzzing）。
8. **定位與診斷分析 (Localization & Diagnosis - FL) ── 🌟 命名一致性微調！**
   - *定義*：分析程式結構或測試軌跡，**計算並輸出 Bug 可能存在的位置或語句可疑度 (Suspiciousness)**。
   - *排除性定義*：排除生成具體補丁代碼或直接修改程式碼的技術（該類歸入修復與補丁合成）。
9. **修復與補丁合成 (Repair & Patch Synthesis - APR) ── 🌟 命名一致性微調！**
   - *定義*：針對已知 Bug 自動生成補丁（Patches）並修改原始碼或二進位。
   - *排除性定義*：排除只定位 Bug 位置而不產出具體修復代碼/補丁的技術（該類歸入定位與診斷分析）。

### 📐 維度二：抽象測試層級 (Abstract Testing Level - Scope)
*核心問句：此技術分析或測試的「最小必要執行環境範圍」是什麼？（排除報告粒度誤導）*

- **單元級 (Unit-level)**
  - *定義*：僅測試獨立的函式、單一類別或孤立 API，其執行不需要依賴完整的應用程式運行環境、資料庫或外部服務。
  - *排除性定義*：排除必須跑完整個專案測試套件、或需要依賴複雜系統服務才能運行的工具（如 GZoltar，雖輸出細到 Line-level，但執行需全專案測試軌跡，歸入系統級）。
- **組件與 IPC 級 (Component / IPC-level)**
  - *定義*：驗證多個組件之間的交互作用，或跨進程通訊接口（IPC，如 gRPC、Binder 接口交互）。
  - *排除性定義*：排除單一系統或單一合約內部的完整狀態空間驗證（該類歸入系統級）。
- **系統級 (System-level)**
  - *定義*：對一個完整運行中的應用程式、內核、全代碼專案或合約系統進行端到端的驗證（如 Certora、GZoltar 等需要全系統或全專案執行環境的技術）。
- **有狀態與多步交互協定級 (Stateful Protocol-level)**
  - *定義*：驗證具有嚴格狀態機約束、需要多個交互步驟（Multi-round exchange）才能完成的網路或加密協定。
  - *排除性定義*：排除無狀態的單一請求-回應檢測（該類歸入組件或系統級）。

---

### 📐 維度三：目標技術棧 (Target Tech Stack / Platform - Where)
*優先判定規則（Precedence Rule）：*
> **優先判斷該研究是否「刻意利用了特定平台/執行環境的底層特徵或 API 機制設計技術」（如：Android Binder 結構、EVM Storage Layout、OS 內核的 syscall table）？若否，一律歸入「通用 (General)」。**

- **通用 (General)**：不綁定特定執行系統，通常僅與程式語言（Java/Python/C等）相關。
- **行動端系統 (Android / iOS)**：與行動 OS 特徵（如安卓組件、Binder）緊密相關。
- **區塊鏈與 Web3 (EVM / WASM / Smart Contract)**：與區塊鏈運行環境及智慧合約相關。
- **作業系統內核 (OS Kernel / Hypervisor)**：與系統內核、虛擬化管理器相關.
- **嵌入式與原生二進位 (Embedded & Native Binary)**：與硬體晶片、韌體或原生桌上型執行檔（如 PE, ELF, Mach-O）相關的底層技術棧。
- **Web 與雲端服務 (Web / API / Cloud)**：與分散式服務、HTTP API 相關。

---

### 📐 維度四：工具與 AI 屬性 (Tooling & AI Attribute - Powered by)
*核心判定原則：*
> **以「核心決策機制 (Core Decision Mechanism)」為準。如果決定「下一步測什麼」、「生成什麼樣的測試/補丁」的核心邏輯是學習出來的模型（包含強化學習的 Reward 決策），就歸 ML/RL；若純靠硬編碼的算法與規則，則歸傳統規則。**

- **傳統規則與啟發式 (Rule-based / Heuristics)**：決策由硬編碼算法、隨機變異、遺傳演算法或 SMT 求解器主導。*排除任何包含訓練/學習決策模型的技術。*
- **經典機器學習/強化學習 (Classic ML / RL)**：使用 GNN、LSTM、強化學習（RL）等做決策。
- **大語言模型與智慧 Agent (LLM / Agent-based)**：核心生成或決策邏輯由 LLM、微調代碼模型或 Agent 規劃鏈驅動。

## 4. 雙層架構下的「複合流派（二級）」與「正交維度（一級）」解耦分析

透過與 39 個實體流派節點對照，可以發現原始分類樹中存在許多「複合型（Composite）」節點。這些節點在命名時，不自覺地將**方法論、平台、與應用場景（Use cases）**焊死在同一個名稱中：
- **`2D.3-智能合約與 Web3 安全`**：物理上它是一個單一節點，但它其實是「區塊鏈/Web3技術棧 + 系統級/合約級驗證 + Fuzzing/符號執行/形式驗證技術」的交織。
- **`1C.2-軟體組成與供應鏈安全(SCA)`**：它是一個應用場景，底層使用「靜態分析（規則/圖比對）」在「通用語言/依賴庫」上進行「單元與組件級」的相依性檢查。
- **`1D.2-密碼學與協議安全審計`**：它是針對「密碼學/協議」這一目標棧的分析，可以使用 AST 匹配、資料流、或形式化驗證等任何底層方法論。

### 核心解耦機制

在雙層架構下，這 39 個實體流派節點**不需要是純粹互斥的方法論**。我們將其定位為**「複合學術流派/應用場景」**：

1. **物理層（39個二級標籤）**：保留其精細的學術名稱與層級分層，作為使用者與訪客最直觀的歸檔入口（例如：「我做 Web3 安全」、「我做供應鏈安全」，這符合學術直覺）。
2. **邏輯層（四維正交一級 Facet）**：在各個複合流派的 Frontmatter 中，使用一級 Facet 進行屬性標記（如數組），來釐清該複合流派在四維正交矩陣中的投影坐標。

這套「雙層解耦」機制既保障了**學術目錄的精細鑑別力**（不砍掉任何一個流派與應用場景），又獲得了**嚴謹、互斥的交叉檢索能力**。

### ⚠️ 二級標籤非互斥性與複合性聲明 ── 🌟 新增！
> **二級標籤（39個實體學術流派）是面向學術研究社群直覺與應用場景的「複合型分類」，其內部節點並不互斥。同一篇論文在標記時，被允許且鼓勵同時連結多個二級標籤（例如同時連結 `[[2D.3-智能合約與 Web3 安全]]` 與 `[[2B.1-反饋引導式模糊測試]]`）。只有一級正交 Facet 屬性（在 Frontmatter 中定義）必須遵守嚴格的正交與互斥定義。**

## 6. 知識庫治理與演進規則

1. **「三票成案」原則（值域穩定性）**：
   只有在知識庫中收集到 $\ge 3$ 篇使用該屬性的獨立論文時，才能在該維度中新增一個子類別或標籤選項。否則，該論文必須先暫時歸類在「通用 (General)」或該維度的父節點下。
2. **殭屍節點合併與降級機制 (Zombie Node Merger & Downgrade) ── 🌟 新增！**：
   如果某個二級物理流派節點（如 `1D.4-網路協定形式化分析`）在知識庫中長期低於 3 篇論文掛靠，為避免知識庫稀疏化，可執行「合併與降級」：
   - **操作**：將該二級流派 note 檔案刪除，將其掛靠的論文物理連結合併至其父類或兄弟類（如 `1D.3-形式化驗證與模型檢查`）。
   - **檢索保護**：被合併的論文，其一級 Facet 屬性（如 `testing_level: "Stateful Protocol-level"`）**保持不變**。未來在首頁通過一級正交篩選時，這篇論文依然能被精確檢索，保障資訊檢索精度不流失。
3. **MECE 定期審查**：
   每半年重新檢視一次首頁儀表板，檢查是否有某個屬性名稱「帶有其他維度的影子」（例如在維度三中出現了名為 "Unit Test" 的標籤），以及檢視維度一中「定位與診斷分析」、「修復與補丁合成」等節點的詞性一致性，一旦發現不一致，立刻進行微調與對齊。
